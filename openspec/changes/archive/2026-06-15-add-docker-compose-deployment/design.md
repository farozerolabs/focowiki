## Context

Focowiki is a pnpm + TypeScript monorepo with three runtime surfaces: Admin UI, Admin API, and public OpenAPI. The existing root `docker-compose.yml` starts PostgreSQL and Redis for local development, while developers run the app with pnpm. Production deployment still requires manual Node/pnpm setup and does not have a repeatable Compose path for application containers, migrations, or static Admin UI serving. S3-compatible storage remains an external dependency configured through environment variables. This change will commit Compose templates only: development users copy `docker-compose.dev.yml.example` to `docker-compose.dev.yml`, deployment users copy `docker-compose.yml.example` to `docker-compose.yml`, and both real Compose files stay local and gitignored.

The deployment design must preserve the existing development commands and keep Focowiki lightweight. The product remains pre-release, so deployment files can be added cleanly without compatibility layers for older container layouts.

## Goals / Non-Goals

**Goals:**

- Add a default Docker Compose deployment path for a self-hosted Focowiki stack through a committed template.
- Rename the current development Compose and env template to development-specific templates.
- Keep real local Compose files out of git.
- Keep the existing pnpm local development workflow unchanged.
- Build application images with multi-stage Docker builds and minimal runtime stages.
- Run Admin UI, Admin API, public OpenAPI, PostgreSQL, and Redis through Compose.
- Keep Admin UI, Admin API, and public OpenAPI reachable through distinct configured high ports, while still supporting reverse proxy deployment.
- Keep raw Markdown and generated files in external S3-compatible storage, while database and Redis state remain in persistent volumes or external services.
- Keep secrets outside images and outside git-visible generated files.
- Provide validation commands for build, startup, migration, health, API, UI, public OpenAPI, and persistence checks.

**Non-Goals:**

- Do not remove or replace `pnpm dev`, `pnpm build`, `pnpm verify`, or the current infrastructure-only Compose workflow.
- Do not introduce Kubernetes, Helm, Swarm, Terraform, or cloud-provider-specific deployment.
- Do not add a heavy process manager or broad runtime abstraction.
- Do not bake `.env`, passwords, API keys, model keys, S3 credentials, or local paths into Docker images.
- Do not add product features unrelated to deployment.

## Decisions

### Decision 1: Commit Compose templates, ignore real Compose files

Move the current infrastructure-only development file from committed `docker-compose.yml` to committed `docker-compose.dev.yml.example`. Add committed `docker-compose.yml.example` as the deployment stack template. Real `docker-compose.yml` and `docker-compose.dev.yml` are local operator files generated by copying templates and MUST be ignored by git. Rename the current local development env template from `.env.example` to `.env.dev.example`, and use `.env.example` as the deployment env template.

The Compose templates should not inline temporary runtime env blocks or fallback defaults for application configuration. Operators copy the relevant env template to the repository root `.env`, and Docker Compose reads that file by default for interpolation and runtime env injection.

Alternative considered: commit real `docker-compose.yml` and `docker-compose.dev.yml`. This was rejected because deployment and development Compose files can carry local ports, paths, overrides, or secrets; the repository should publish templates only.

### Decision 2: Use one root Dockerfile with named runtime targets

Use a root monorepo Dockerfile with multi-stage targets:

- dependency/build stages install pnpm dependencies once from lockfile and build workspace packages.
- an API runtime target contains compiled API JavaScript, compiled workspace package output, production dependencies, migrations, and no dev-only test files where practical.
- an Admin runtime target contains only the built Admin UI static assets plus a small static server or reverse-proxy image configuration.

Alternative considered: separate Dockerfiles under each app. This was rejected because both app images need monorepo workspace context and would duplicate dependency/build logic.

### Decision 3: Serve Admin UI as static production assets

The deployment Admin UI container must serve `apps/admin/dist` instead of running Vite dev server. A minimal static server image such as `nginx:alpine` is acceptable if it is configured with SPA fallback, security headers, and optional `/admin/api` proxy behavior.

Alternative considered: run `pnpm --filter @focowiki/admin dev` in containers. This was rejected because Vite dev server is not the production deployment target.

### Decision 4: Keep API and public OpenAPI in one API process for now

The current API process starts Admin API and public OpenAPI from one Node process and listens on two configured ports. The Compose deployment should preserve that architecture for this slice: one `api` service exposes both `ADMIN_API_PORT` and `PUBLIC_OPENAPI_PORT`.

Alternative considered: split Admin API and public OpenAPI into separate containers. This may be useful later, but it would require a new runtime entrypoint and more coordination for a deployment change that should stay minimal.

### Decision 5: Use an explicit migration service

Compose deployment should include a one-shot `migrate` service that runs `pnpm --filter @focowiki/api db:migrate` or the runtime equivalent against the same API image. The `api` service should depend on healthy PostgreSQL, Redis, and completed migrations where Compose supports it. S3-compatible storage is validated by runtime configuration and by upload/openAPI validation against the configured external endpoint.

Alternative considered: run migrations inside the API startup command every time. This was rejected because it hides deployment state and makes rollback/restart behavior less explicit.

### Decision 6: Require external S3-compatible storage

Compose deployment should not start MinIO or another object store. Operators must provide an external S3-compatible endpoint and credentials through `S3_ENDPOINT`, bucket, region, credentials, prefix, and path-style mode. This keeps deployment Compose focused on the application, PostgreSQL, and Redis, and avoids exposing or operating object storage as part of the app stack.

### Decision 7: Keep production security explicit

The default deployment env template `.env.example` must require production-safe secrets and origins: `APP_ENV=production`, strong `ADMIN_SESSION_SECRET`, HTTPS public origins for real public deployment, `ALLOWED_HOSTS`, trusted origins, cookie settings, public API auth, S3 credentials, and model config when enabled. Compose examples may use loopback HTTP only for local deployment validation.

Alternative considered: ship weak default production credentials. This was rejected because the app is intended to be publicly deployable behind a reverse proxy.

## Risks / Trade-offs

- [Risk] Admin UI build-time `VITE_ADMIN_API_BASE_URL` can make one image environment-specific. -> Mitigation: support same-origin proxy mode where practical, or document/build with explicit build args for separated Admin API origins.
- [Risk] Workspace packages may resolve TypeScript source paths at runtime if package exports are not production-compatible. -> Mitigation: validate the API image runs from compiled JavaScript and includes compiled workspace package output or production-ready package exports.
- [Risk] Minimal runtime images can miss migration files or runtime assets. -> Mitigation: add container tests that run migrations and start API from the image.
- [Risk] Compose health checks can pass while external S3 is unreachable. -> Mitigation: runtime upload validation and release-gate tests must cover configured external S3 before production use.
- [Risk] Reverse proxy deployments differ from local loopback validation. -> Mitigation: document both local Compose validation and production reverse proxy settings.
- [Risk] Image size can grow if dev dependencies or build cache are copied into runtime targets. -> Mitigation: use multi-stage copy allowlists and inspect image contents/size during validation.

## Migration Plan

1. Add `.dockerignore` to exclude git metadata, local env files, node_modules, build cache, OpenSpec reports, local data, and test output from Docker build context.
2. Add root Dockerfile with multi-stage build targets for API runtime and Admin static runtime.
3. Rename the current local development `docker-compose.yml` to `docker-compose.dev.yml.example` and update local development docs and tests.
4. Rename the current local development `.env.example` to `.env.dev.example` and update local development docs and tests.
5. Add a new default deployment template `docker-compose.yml.example` for the application deployment stack, using persistent volumes for PostgreSQL and Redis while requiring external S3-compatible storage.
6. Add a deployment env template at `.env.example` without real secrets.
7. Add `.gitignore` entries for real local `docker-compose.yml` and `docker-compose.dev.yml`.
8. Add package scripts for deployment build/start/stop/logs/migrate/validation where useful.
9. Add tests that assert Compose templates define services, health checks, high ports, no memory-only storage, no dev server in production, and no secrets in images or docs.
10. Update README deployment documentation with template copy steps, local Compose deployment, and reverse-proxy deployment guidance.
11. Validate the Compose templates with `docker compose -f docker-compose.yml.example config` and `docker compose -f docker-compose.dev.yml.example config`, then validate copied local Compose files during runtime smoke checks.

Rollback is normal git revert plus `docker compose down` for local deployment containers. Pre-release local volumes may be destroyed with `down -v` when needed.

## Open Questions

- None. Use the recommended path: publish only `docker-compose.yml.example` and `docker-compose.dev.yml.example`, ignore real local Compose files, use a root multi-target Dockerfile, keep one API container exposing Admin API and public OpenAPI, serve Admin UI as static assets, and require external S3-compatible storage through env configuration.
