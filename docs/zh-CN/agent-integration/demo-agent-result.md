# Demo 运行测试结果示例

这个示例展示第三方 Agent 通过 Focowiki demo 服务读取法律知识库，并基于知识库内容回答问题。

Demo 服务把 Focowiki OpenAPI key 保存在后端。Skill 通过 HTTP 请求访问 demo 服务，搜索已配置的知识库，读取命中的文件，并基于读取到的证据生成回答。

## 调试结论与研究方向

经过多轮真实法律文档调试，Focowiki 在探索深度和回答准确性上，相较传统 RAG 知识库有了明显提升。关键原因是知识库以文件为第一组织方式，同时提供索引、元数据、图关系文件、关联链接和可直接阅读的页面文件。Agent 可以检查资料库结构，扩展线索，读取完整文件，比对证据，并在回答中引用来源。

当前瓶颈已经从知识切片和召回组合，转向 Agent 如何围绕知识库组织 Loop。浅层 Loop 通常只读取一两个文件，回答容易变窄。完整 Loop 会从问题出发，拆解检索短语，读取 index 和 schema，打开相关文件，从正文里抽取新的线索，再继续检索和比对，最后收敛成答案。当 Loop 组织完整时，回答会更精确，也更完整。

我们最开始用 RAG 思路验证搜索，发现 chunk 召回容易漏掉文档整体上下文、跨文件关系、更新状态和法律结构。经过我们的探索，我们放弃了传统的 RAG 知识库。由于 RAG 知识库的缺陷，在不同的数据中需要进行嵌入、重排、切分片段的调试，这让调试过程非常繁琐，也不太容易保证稳定性和可探索性。随后把核心表示迁移到 Markdown 文件，保留元数据，生成索引文件，增加图关系和关联链接，并设计 Skill 引导 Agent 循环探索。这个方向让知识库同时便于人类审计和 Agent 导航，也让最终推理回到 Agent 的探索能力上，而不依赖固定切片来表达全部知识。

## 这个示例说明什么

- Skill 可以引导 Agent 在回答前先查询配置好的知识库。
- Demo 服务可以为第三方 Agent 客户端提供一个小型只读 HTTP 接口。
- Agent 可以从搜索结果继续读取文件，并基于证据生成回答。
